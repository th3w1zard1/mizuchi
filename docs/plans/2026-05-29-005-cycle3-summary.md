## Cycle 3 Architecture Highlights

### State Machine

```
Prompt Folder (prompts/<name>)
         ↓
    [pending] ← scored functions
         ↓
    Matcher.sh (one-shot Claude)
         ↓
    trial.c generated
         ↓
    Build-and-Verify.sh
    ├─ Compile (defensive output cap)
    ├─ Objdiff check
    └─ Commit if verified
         ↓
    [matched] → git commit + queue update
         ↓
    User: `/decomp-integrate` (separate step)
         ↓
    [integrated] → source tree landed

If objdiff fails:
    [failed] ← attempt count incremented
         ↓
    If attempt_count >= 10:
        [difficult] ← flagged for manual triage
    Else:
        Back to [pending] for retry
```

### Vacuum Loop Flow

```
vacuum.sh (orchestrator)
    ↓
Initialize: load queue, run scorer
    ↓
LOOP:
    1. Load queue from JSON
    2. Get next pending (highest score)
    3. Call matcher.sh (one-shot Claude)
    4. Call build-and-verify.sh
    5. Update queue state
    6. Handle backoff if quota hit
    7. Continue or exit
    ↓
Exit: Save state, produce summary
```

### Token Efficiency

- **Prompt:** Fixed template (no in-prompt history), context injected at startup
- **Build output:** Capped (first error + last 5KB stderr)
- **Logging:** Dual-channel (human-readable progress log + debug file)
- **Backoff:** Exponential (5s, 10s, 20s, ..., 60min) on 429/timeout

### Persistence & Safety

- **State:** JSON files in `state/` (queue.json, scores.json, attempt_log.json)
- **Atomic writes:** Temp file + mv prevents corruption
- **Signal handling:** Ctrl-C trap, graceful shutdown + state save
- **Commits:** One per verified function (rollback-safe, traceable)
